# 提問策略（Clarifying Questions）

## 何時需要問

只有在「不問就會導致結果差異很大」或「涉及風險/不可逆」時才問。

常見必問情境：

- 需求含糊導致不同實作方向（例如 UI 風格、資料模型、權限範圍）。
- 會改寫歷史/刪資料/影響線上（例如 force push、刪 DB）。
- 需要使用者提供不可推得的值（API key、帳號、環境限制）。

## 怎麼問（只問一題）

原則：

- 先做不會被問題卡住的工作（例如讀文件、列出選項）。
- 問 **一個**最關鍵的問題。
- 給「建議預設」並說明不選預設會差在哪。

模板：

```text
我現在缺一個關鍵資訊：<問題>
我建議預設選 <A>（原因：...）。
如果選 <B>，差異是：...
```
